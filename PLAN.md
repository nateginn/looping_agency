# Plan: SEO monitoring & reporting expansion for `art` (agency-grade)

_Locked via grill — by Claude + Nate. Supersedes the prior Phase 3 plan below in this file's history — see git log for the auto-implement/verify-loop plan (shipped 2026-07-21, commit `59f40f8`), which remains live and unaffected by this plan._

## Goal

Extend the existing `art`/`seo` loop so it monitors the site the way a dedicated SEO agency would: local/geo keyword ranking by clinic location (zip code or a 20-mile radius around the exact address), backlink profile health, and free technical/on-page signals (Core Web Vitals, indexation status) — on top of the GSC query/page data it already pulls. All of it lands inside the loop's existing run artifacts (`run.json` / `report.md` under `projects/art/loops/seo/runs/<run-id>/`) so it reaches Nate through the MMC daily Telegram briefing with **zero changes inside `D:\Dev\MMC`** — MMC already reads those files and just gets richer content to summarize. This is monitoring/reporting only: no new proposal types, no new `allowed_actions`, no change to the approval pipeline. Ads and content-social loops are untouched.

## Approach

0. **Capability spike before any connector code is written.** Nate pulled DataForSEO's actual API catalog: **Business Data API → Google My Business** is the named candidate for hyperlocal (zip/20-mile-radius) rank tracking — distinct from the SERP API's organic-web rank-by-`location_code` already wired in `tools/dataforseo.py`, which only supports broad city/state-level targeting, not arbitrary zip/radius precision. The spike must resolve, against Nate's actual DataForSEO account/plan: (a) that Business Data API → Google My Business is on his plan and does what's needed, (b) its request/response shape and any distinct auth/pricing implications versus the SERP endpoint already in use, and **(c) — critically — which metric it actually measures.** Google My Business rank is very likely a **Local Pack/Maps position** (the business listing's rank among nearby competitors, roughly 1-3 shown in the map pack, not tied to a specific page) — a fundamentally different signal from **organic page rank** (a URL's 1-100 position in web search results, what GSC/the existing SERP function measure). If step 0 confirms it's Local Pack rank rather than organic rank, **it must be modeled as a distinct metric family with its own schema and thresholds** (step 6's `local_rank` snapshot section, step 5's rank `numeric_delta` threshold) — not silently reused as an extension of the existing `rank_position`/`baseline_position` concept, which would corrupt reporting and thresholds by comparing unlike signals. **Fallback if unsupported or not on-plan: reduce geo-rank scope to city/state-level `location_code` organic targeting** via the existing SERP endpoint. The plan's headline goal is not implementable as written until (a)-(c) are resolved.

1. **New connectors, each a separate function with its own tests — not folded into existing ones** (injected credential resolver, redaction, `--verify` self-test, matching `gsc.py`/`dataforseo.py`'s existing pattern):
   - New location-aware rank-tracking function targeting **Business Data API → Google My Business** per step 0's outcome (or the SERP API's `location_code` as fallback) — one call per configured clinic location — plus a separate new `pull_backlinks()` using DataForSEO's **Backlinks API** (referring domains, total backlinks, new/lost since last check). Both likely belong in `tools/dataforseo.py` alongside the existing SERP function, but Business Data API being a genuinely different product family (not just a different endpoint) may warrant its own module (`tools/dataforseo_business.py`) — a call to make once step 0's actual request/response shape is known.
   - New `tools/pagespeed.py` — Core Web Vitals (LCP/INP/CLS, performance score) via Google's free PageSpeed Insights API for each priority page (see step 2a for where that page list comes from). Noted alternative, not adopted: DataForSEO's own **OnPage API → Lighthouse** could produce the same CWV data on the same vendor/invoice as everything else — staying with free PSI unless Nate would rather consolidate onto one vendor.
   - New, separate functions in `tools/gsc.py` for indexation status — Sitemaps resource (submission/processing status) and URL Inspection API (quota-limited, ~2,000 calls/day/property, scoped to priority pages only) — kept distinct from the existing Search Analytics function since they're a different endpoint family, auth assumption, and quota regime, not a small additive change to `pull_metrics`.

2. **A single connector registry drives both validation and dispatch — not two lists that can drift.** Today `spec_validate.py`'s `KNOWN_INPUTS` and `run_loop.py`'s `_fetch_metrics` dispatch are separate hardcoded lists; a spec can validate against `KNOWN_INPUTS` and then fail at run time with "no connector wired" if `_fetch_metrics` wasn't also updated. Fix this as part of adding the new connectors: one registry (module/dict) mapping connector name → {validation requirements, dispatch function, required credential alias}, imported by both `spec_validate.py` and `run_loop.py`. Add `dataforseo-backlinks` and `pagespeed` to that registry (e.g. `pagespeed` requires a priority-pages list; `dataforseo-backlinks` requires the domain).

   **2a. Priority pages become machine-readable config, not prose.** `project.md`'s current "Priority reference pages" section is explicitly documentation-only (a human review aid, not parsed by any tool). CWV and URL Inspection checks need an actual list to iterate — add a `priority_pages` array to `spec.md`'s frontmatter (validated, like `keyword_exclusions` already is) as the machine source of truth; `project.md`'s prose section stays as-is for human context and isn't scraped.

3. **Locations, same treatment as priority pages: validated `spec.md` frontmatter, not `project.md` markdown.** Add a `locations` array to `spec.md`'s frontmatter (name, address, zip — validated by `spec_validate.py`, one entry per clinic, not hardcoded to today's two) as the actual machine source of truth that runs read from. `project.md` may still carry a human-readable `## Locations` note for context, but it is documentation only and never parsed — this repo has no parser for structured markdown-body sections, and treating one as authoritative would repeat exactly the priority-pages prose problem just fixed in 2a. Once a month (folded into that week's Monday full run, no new schedule entry), a read-only fetch of the live site's public footer HTML checks for address changes and writes any detected diff to a generated file (e.g. `projects/art/loops/seo/locations-detected.json`) plus a flagged line in that run's `report.md` — **it never auto-edits `spec.md` directly.** Nate reviews the flagged diff and updates `spec.md`'s `locations` array himself if it's correct, the same way he already reviews and decides on proposals. Runs read the *confirmed* `spec.md` list, not the detected-but-unconfirmed file.

4. **Two schedules on the same loop, with technical-only runs kept structurally separate from proposal logic.** Both write into the same `runs/` directory so MMC's "latest run" logic keeps working unmodified:
   - **Monday 06:00 — full run:** existing GSC pull + geo-rank (all locations) + backlinks + technical (CWV + indexation) + the monthly footer check on the first Monday of the month.
   - **Thursday 06:00 — technical-only run:** CWV + indexation only, no DataForSEO calls, no GSC Search Analytics pull (keeps paid API usage weekly, not twice-weekly).
   `spec.md` frontmatter gains an `additional_schedules` list (name + cron + inputs subset); `run_loop.py` gains a `--run-name <name>` flag. **Each `additional_schedules[*]` entry is validated as its own fully resolvable runnable mode** — `spec_validate.py` checks its `inputs` subset against the connector registry (step 2) exactly like top-level `spec.inputs`, including that every connector it names has a required credential alias present, not just that the subset is well-formed. A spec with an `additional_schedules` entry referencing an unregistered connector or a missing alias fails validation before any run, same as today's top-level check. **`run_loop.py` must treat `mode: "technical-only"` as observe-only and skip `_evaluate_prior_experiments` and `_pick_new_actions` entirely** — those functions assume a GSC-shaped metrics payload (`keywords`, `position`, `sample_size`) that a technical-only run never produces; running them unconditionally would break on missing fields, not just produce an empty result. `run.json` records `mode: "full" | "technical-only"`.

5. **Threshold-based "needs attention" flags require a sustained/material delta, not a single-run blip, to avoid paging Nate on routine noise.** New `attention_thresholds` block in `spec.md` with its **own explicit schema** — distinct from `guardrail_metrics` (which is numeric comparator/threshold/`consecutive_runs` only, per `spec_validate.py`'s `_validate_guardrail_metric`, and can't represent a categorical state change) — each entry typed by `kind`:
   - `numeric_delta` — `{kind: numeric_delta, metric: <resolved by step 0(c) — either "organic_rank_position" or "local_pack_position", never both under one name>, comparator: ">", threshold: 5, consecutive_runs: 2}`: rank drop > 5 positions at a tracked location, held for 2 consecutive runs before flagging. The metric name and its threshold's meaning depend entirely on step 0(c)'s finding — a 5-position drop means something very different for a 1-100 organic rank than for a 1-3 Local Pack slot, so the concrete metric name (not a generic "rank_position") and threshold value get finalized once step 0 resolves which signal this actually is.
   - `numeric_delta` — `{kind: numeric_delta, metric: referring_domains, comparator: "<", threshold: -3, consecutive_runs: 2}`: concrete default of losing 3+ referring domains, sustained across 2 runs (replaces the earlier unresolved "any decrease" / placeholder `>= N`).
   - `enum_transition` — `{kind: enum_transition, metric: cwv_status, to: poor}`: fires the run a priority page's CWV status is observed transitioning into "poor" — a discrete state change, not a noisy metric, so no `consecutive_runs` field applies or is accepted for this kind.
   `spec_validate.py` validates each entry's shape by its `kind` (numeric fields required for `numeric_delta`, `to` required for `enum_transition`, `consecutive_runs` rejected on `enum_transition` entries). Crossed thresholds get a dedicated section at the top of `report.md`, matching the "Needs attention" framing MMC's briefing already uses today. This never touches `pending/`, `state.json`'s breach mechanism, or the approval state machine — it's Tier 0, purely observational.

6. **Richer `report.md`, each section stamped with its own as-of date** (since technical data now updates twice weekly and geo-rank/backlinks weekly, within the same file): Needs Attention, Local Rank by Location, Backlinks, Technical Health (CWV + indexation), plus the existing GSC/proposals sections unchanged. **`run.json` stays lightweight** — new connectors add call-metadata entries to `tool_calls` only (tool name, args, timestamp, ok/fail), matching today's GSC/DataForSEO entries exactly. **`snapshot.json` gains a versioned, sectioned schema** rather than an ad hoc merge: top-level keys `search_analytics` (today's GSC data, unchanged shape), `local_rank`, `backlinks`, `technical_health`, each carrying its own `as_of` timestamp and its own internal schema. **`local_rank`'s internal schema is not assumed to look like `search_analytics`'s** — per step 0(c), if the underlying signal is Local Pack/business-profile rank rather than organic page rank, it gets its own field names (e.g. `local_pack_position`, no associated `page` field, since a Maps listing rank isn't tied to a URL the way organic rank is) and its own section in `report.md`'s "Local Rank by Location," kept visually and structurally distinct from organic position data rather than presented as the same kind of number.

7. **Document, don't register, the new Task Scheduler entry.** Follow the existing convention in `PHASE3-SCHEDULING.md` (ready-to-run `schtasks` commands, never executed by tooling) — add the Thursday technical-only job alongside the Monday full-run job already documented there.

8. **Verify the MMC delivery assumption with a real artifact before calling it done.** After a live run produces a richer `report.md`, manually trigger (or wait for) one MMC daily briefing cycle and confirm the Telegram output actually surfaces the new sections coherently. This is a verification step against MMC's *existing, unmodified* behavior — not a code change inside `D:\Dev\MMC` — but "reaches Nate automatically" shouldn't be treated as confirmed until it's been observed once for real.

## Key decisions & tradeoffs

- **DataForSEO for geo-rank + backlinks, free Google APIs for technical health** — reuses the credential/connector pattern already in this codebase instead of adding a third vendor; avoids paying for Core Web Vitals/indexation data Google gives away free.
- **A capability spike (step 0) gates the hyperlocal rank-tracking goal before any code is written**, with an explicit city/state-level fallback if zip/radius precision isn't available on Nate's account — rejected treating this as a footnote risk to discover mid-implementation, since it determines whether the plan's headline goal is achievable as stated.
- **Validation and dispatch share one connector registry**, replacing today's two-hardcoded-lists design (`spec_validate.py`'s `KNOWN_INPUTS` vs. `run_loop.py`'s `_fetch_metrics`) that can silently drift — a spec passing validation must actually be runnable.
- **Priority pages move into `spec.md` as validated frontmatter, not left as `project.md` prose** — the existing "Priority reference pages" section is explicitly a human review aid today, not machine-readable; CWV/indexation checks need real structured input, not markdown-bullet scraping.
- **Everything stays inside the existing `seo` loop's artifacts** rather than a new sibling loop — a deliberate constraint so MMC needs zero changes; the tradeoff is `report.md` now mixes data of different freshness (weekly vs. twice-weekly), handled with per-section as-of stamps rather than a second artifact path. `run.json` stays call-metadata-only (as today) precisely so this richer content doesn't bloat the artifact MMC's summarizer reads in full.
- **The monthly footer scrape writes to a generated file and flags the diff — it never auto-edits `project.md` directly.** Rejected auto-rewriting the human-reviewed location list: that would let a scrape silently redefine authoritative config, dirty the repo from an unattended run, and race with any manual edit Nate makes in between. Human confirmation stays required, matching how every other proposal in this system already works.
- **Pure monitoring, no new proposal types** — keeps this plan's scope to observability only, and deliberately doesn't compound the existing gap (found this session) where today's proposals lack real proposed-copy content. That's a separate, pre-existing problem this plan does not attempt to fix.
- **Two schedules, one loop** rather than a fully separate technical-health loop — costs a small schema addition (`additional_schedules`, `--run-name`) but keeps one report, one memory.md, one place Nate looks. Technical-only runs are explicitly observe-only and bypass `_evaluate_prior_experiments`/`_pick_new_actions`, so the proposal-generation path never runs against a payload it wasn't designed for.

Flagging for Nate's expectations, not blocking the plan.

Nate stores these himself, aliases only in spec.md, per existing convention.
- **Whether MMC's Telegram briefing actually surfaces the new report sections usefully is unverified until a live run** — addressed by Approach step 8's explicit post-implementation check, not left as an unverified assumption.

## Goal

Phase 2 is complete: the `art` project (acceleratedrehabtherapy.com) runs a live, propose-only SEO loop against real Google Search Console data (54/54 exit-criteria checks passing, real runs producing real proposals). The loop currently dead-ends at human approval — nothing ever implements a proposal, and the existing verification code that would measure "did this change improve ranking" (`_evaluate_prior_experiments` in `tools/run_loop.py`) never fires because no proposal ever reaches `status: "applied"`.

Phase 3 closes that loop: once a human approves a proposal, an agent writes the actual code change as a commit on a local branch inside the sibling website repo (`D:\Dev\artwebsite`) — but never pushes. A new read-only check detects when the human has pushed it live (reusing a pattern MMC already uses against this same repo: compare local HEAD vs. GitHub API for `origin/main`, no fetch). Only then does the proposal become `applied`, which unblocks the dormant verification logic to measure real before/after Google position.

## Approach

1. **Policy scope correction (must land first, before any code).**
   - `RISK-REGISTER.md` R6 currently reads, in its resolution: "Phase 1 does not touch this repo... at all" and requires confirming "no run can write into that repo's working directory." This is stricter than the actual rule the rest of the doc set states (`AgentColabPlan.md`'s Tier 2 definition scopes specifically to `git push`, not to every write). Amend R6's resolution text to explicitly scope the restriction to `push`/`merge`/remote-tracking-branch operations — a local working-tree commit/branch is not itself Tier 2, but pushing it always is, unconditionally.
   - Rewrite `projects/art/project.md` and `projects/art/loops/seo/spec.md`'s rollback text, which currently states this loop's tooling "does not currently read or write anything inside `D:\Dev\artwebsite`, ever" — that line becomes false under this plan and must be corrected to describe the new, narrower boundary (commit/branch only, never push).
   - `D:\Dev\artwebsite\CLAUDE.md` needs **no change** — it already only forbids automated *push* ("No automated tool ... may push here"), which is compatible with this plan as designed.

2. **Auto-implement on approval — extend `tools/apply.py`.**
   - After the three existing checks (`tier != 2`, `approval_mode == "tier1-enabled"`, `status == "approved"`), add a fourth, hard, unconditional check: **`manual_approval_only: true` on the proposal always refuses apply, regardless of `approval_mode`.** This is a firm decision, not a documentation-only flag anymore — a spec author must explicitly remove it per-action to opt that action into auto-implementation. Only then dispatch to a new per-`action_type` implementer.
   - **Scope this phase to `title-tag-rewrite` and `meta-description-rewrite` only.** `internal-link-addition` has no deterministic, safe edit strategy (which existing link, which anchor text, which target URL to pick is a content judgment, not a mechanical rewrite) and stays `manual_approval_only: true` permanently for now — auto-implement is not offered for it until a tested content-edit contract exists for that action type specifically.
   - The implementer resolves each target page to its real edit location **at runtime, not from a static hardcoded map** — it opens the actual page's Django view/template, checks whether a `{% block title %}`/`{% block meta_description %}` override exists; if not, falls back to the `views.py` context-dict pattern. If neither location is found in the expected shape, it **fails closed** with a clear, surfaced error (new proposal status `implement-failed`, human-readable reason) rather than guessing or silently no-op'ing. (Repo recon informs what to expect — home/`auto-injury` use the `views.py` pattern, other pages use template blocks, per a real prior commit `6e2ad77` that fixed a silent-shadowing bug from this exact dual pattern — but the implementer must verify this at execution time since recon can go stale.)
   - **Writes to an isolated `git worktree`, never Nate's own checked-out working tree.** `apply.py` runs `git worktree add <tmp-path>/seo-<proposal-id> -b seo/<proposal-id> main` inside `D:\Dev\artwebsite`, makes the edit and commit inside that separate worktree directory, then `git worktree remove`s it — Nate's actual working directory (which may be dirty, mid-merge, or on another branch at any time) is never touched or required to be clean. This closes the working-tree-hygiene gap from Round 2 review more completely than requiring a clean tree would.
   - **Crash-recoverable by construction, and recovery must not destroy real work (Round 3 correction):** before creating the worktree/branch, `apply.py` writes an `implement_attempt: {started_at, worktree_path, base_commit}` record onto the proposal (via the same locked atomic write below) — this happens *before* touching git. If `apply.py` crashes mid-implementation, the next invocation (or a small cleanup check at the top of `apply_proposal`) finds an `implement_attempt` with no matching `implemented_commit_sha` and **inspects the recorded branch before doing anything else**: if the branch HEAD still equals `base_commit`, no work was done — safely remove the stale worktree/branch and reset to `approved` for a clean retry. If the branch HEAD has moved past `base_commit`, a valid commit exists — **promote it** (write `implemented_commit_sha` = that HEAD, `implemented_branch`, `implemented_at`, transition to `implemented`) instead of deleting it. The original Round 2 design would have silently discarded a completed implementation if the crash landed between "commit created" and "JSON updated" — this is now impossible by construction, not just less likely.
   - New proposal status: `implemented` (not yet `applied`) — records `implemented_branch`, `implemented_commit_sha`, `implemented_at`. On failure (page-location mismatch, git error): status `implement-failed` with a human-readable `implement_error` field. **Recovery path:** `review_pending.py` gains a new `--retry <id>` transition, `implement-failed → approved` — the human reads the error, fixes the underlying cause if needed, and explicitly re-queues the same proposal for another attempt. This is the missing recovery path flagged in Round 2; without it, `implement-failed` was a dead end since `apply.py`'s existing gate only accepts `status == "approved"`.
   - **Concurrency/locking uses a dedicated apply-scoped lock, not the run-cadence lock (Round 3 correction).** The existing per-loop lock (`tools/lib/lock.py`) is age-expiring against `max_run_duration_minutes` and designed for scheduled `run_loop.py` runs — an implement operation (worktree create, edit, commit, possible cleanup) can legitimately run longer or on a different rhythm, and reusing that lock risks a scheduled run "recovering" it as stale mid-implementation and mutating the same proposal files concurrently, which defeats the point of locking at all. Instead: a second, separate lock file scoped to `apply`/implement operations for that loop (same primitive from `tools/lib/lock.py`, different lock name/file, its own TTL sized to a worst-case implement attempt, not a run). `review_pending.py`'s new `implemented`-status write path (the live-detection check in `run_loop.py`, step 3) also acquires this apply-scoped lock — briefly, not for its whole run — whenever it mutates a proposal's status, so the two lock domains (run-cadence vs. implement-in-progress) never contend on each other's TTL assumptions. All writes still use atomic temp-file-replace.
   - `art`'s `spec.md` must have `manual_approval_only: true` removed from the two in-scope actions and `approval_mode` flipped to `tier1-enabled` before this path can activate for `art`; this is a deliberate, separate switch the human flips when ready, not implied by shipping the code.

3. **Detecting the change actually went live — new, small, read-only.**
   - Reuse the pattern already documented in `D:\Dev\MMC\registry\systems.yaml` for `artwebsite`'s own status check, made concrete: GitHub's compare API, `GET /repos/nateginn/artwebsite/compare/{implemented_commit_sha}...main`. If `status` is `"identical"` or `"ahead"`, `implemented_commit_sha` is an ancestor of `main` — the change is live. Read-only, **never `git fetch`** into that repo.
   - **applied_at is set to the time this check first detects ancestry (i.e., "now," at detection), never the commit's own author/committer timestamp.** Commit time is when the branch was written, not when it went live — using it would start every observation window too early and corrupt every verified/breached result. This was the single most important correction from Round 1 review.
   - **Documented limitation, not silently swallowed:** this detection contract only recognizes a fast-forward or regular (non-squash) merge of the exact branch. A squash-merge or rebase produces a new SHA not present in `main`'s history via simple ancestry, so detection will never fire for it. This is an explicit operational constraint communicated to the human (merge normally, don't squash) rather than a hidden gap.
   - **Visibility is immediate, escalation is graduated (Round 2 correction):** every run, any proposal still in `implemented` status is listed in `report.md` under "Awaiting live confirmation" starting from the very first cycle it's not yet detected — not held back until some threshold. Only after 3+ run cycles without detection does it additionally get flagged `stuck-implemented` (a distinctly urgent marker suggesting the human check whether a squash-merge broke detection). With the weekly cadence, surfacing only at 3 cycles would mean a real detection failure could sit invisible for ~3 weeks — Round 2 flagged this as too slow for what is, in the squash-merge case, actually a silent execution-state failure, not a slow-moving metric.

4. **New status integration — explicit, not implied.** Adding `implemented`/`implement-failed` touches multiple existing consumers; each must be updated deliberately:
   - `tools/review_pending.py`'s `TRANSITIONS` dict gains exactly one new entry: `"retry": {"from": ["implement-failed"], "to": "approved"}`. `implemented` itself is still never human-reachable via `--approve`/`--reject`/`--review` — only `apply.py` sets it. `--list` already prints whatever `status` field is present generically, so no further change needed there.
   - `run_loop.py`'s cooldown/in-flight tracking (currently `still_cooling_down` only tracks `status == "applied"` targets, `run_loop.py:415-417`) must be extended to also exclude targets with a proposal in `approved`, `implemented`, or `implement-failed` status — otherwise the loop keeps generating new draft proposals against a page that already has an approved-or-in-progress change queued, producing branch conflicts and contradictory recommendations. This is a concrete gap in the current code, not a future nice-to-have.
   - `_evaluate_prior_experiments`'s stale/cooldown logic stays scoped to `applied` only, unchanged — it is the correct gate for verification and needs no modification.

5. **Verification loop — no new logic.** `_evaluate_prior_experiments` (`run_loop.py:234-282`) already does exactly the before/after comparison needed (`baseline_position` vs. current GSC position, `verified`/`breached` outcome, guardrail check) — it is simply never reached today because nothing hits `status == "applied"`. Steps 2-4 make that condition reachable; no changes to this function's own logic are planned.

6. **Report / dashboard surfacing.**
   - Extend `report.md` generation to list proposals by status each run, and for any newly `verified`/`breached` proposal this run, show the position delta explicitly (already present in the `decisions` list text, just needs to also land in `report.md`'s body, not only `run.json`).
   - `D:\Dev\MMC\registry\systems.yaml` already reserves a `status_artifacts` slot for Looping Agency's `pending/` and `run.json`/`report.md` — confirm (separately, in MMC's own repo) whether its daily-briefing collector actually reads this yet; if not, that's a small MMC-side change, out of scope for this plan's file edits but noted as a dependency.

7. **Scheduling.** Register a Windows Task Scheduler job running `tools/run_loop.py art seo` on the cadence already declared in `spec.md` (`0 6 * * 1`, weekly Monday), following the same pattern as the existing `ARTDailyReport`/`MMCDailyBriefing` tasks already on this machine. A second, independent task runs `tools/watchdog.py` to catch silent scheduling failures (already built, never wired to a scheduled trigger).

## Key decisions & tradeoffs

1. **Commit-not-push as the boundary, rather than a fuller PR-based flow.** `D:\Dev\artwebsite` has no staging environment and no PR/CI-preview flow — the generic loop template's default `rollback: revert PR` pattern doesn't map cleanly here. A local unpushed branch is the closest analogue: reviewable, revertible (delete the branch), zero live effect until a human separately pushes.
2. **`manual_approval_only: true` is now a hard, unconditional apply-time refusal in `apply.py`** — firm decision (Round 1), not documentation-only. A spec author opts a specific action into auto-implementation by removing the flag from that action only; other actions on the same loop can stay manual. This is also why `internal-link-addition` can stay permanently manual (approach step 2) without needing a second loop.
3. **`applied_at` is the detection time of the live-check (step 3), never a commit timestamp.** Corrected in Round 1 — using commit-to-branch time would start every observation window before the change was actually live, corrupting every verified/breached result.
4. **Detection contract is exact-ancestry via GitHub's compare API, not content-diff matching.** Simpler and unambiguous, at the cost of not recognizing squash-merges/rebases — an explicit, documented operational constraint (approach step 3) rather than a silently-missed case.
5. **No dashboard is built in this phase.** A dashboard/report is only meaningful once real `verified`/`breached` data exists; building it before then would just render an empty state. `report.md` gets richer, but a dedicated dashboard is deferred until at least one real verify/breach cycle has happened.
6. **Proposal/state mutations require the per-loop lock and atomic writes (approach step 2).** Closing this now rather than deferring it — the auto-implement path is exactly the kind of new concurrent writer (manual `apply.py` invocation vs. a scheduled `run_loop.py` run) that turns a latent race into a real one.
7. **Isolated `git worktree` per proposal, not Nate's own checked-out working tree (Round 2).** Rejected the simpler "require a clean tree" fix — a worktree makes the working-tree-hygiene problem structurally impossible instead of just checked-for, and never blocks or interferes with whatever Nate happens to be doing in his own `artwebsite` checkout at the time.
8. **`implement_attempt` metadata written before any git operation, not after.** This is what makes a mid-implementation crash recoverable: the next run can always tell "was a worktree/branch started for this proposal, and did it ever produce a commit" and clean up deterministically, rather than guessing from git state alone.
9. **`implement-failed` gets an explicit human-gated retry (`review_pending.py --retry`), not an automatic one.** An automatic retry could repeat the same failure indefinitely (e.g. a genuinely-changed page structure); requiring the human to look at `implement_error` first, then explicitly re-queue, keeps the human in the loop for the failure path the same way approval is required for the happy path.
10. **`stuck-implemented` visibility starts at cycle 1, escalation flag at cycle 3 (Round 2 correction).** Waiting until cycle 3 (~3 weeks at the weekly cadence) to show anything at all was too slow for what can be a genuine execution failure, not a slow-moving metric — the report now shows "awaiting live confirmation" immediately and only adds the urgent flag later.
11. **Crash recovery inspects git state before ever deleting a branch (Round 3 correction).** The Round 2 design reset-to-`approved`-on-any-crash was unsafe: a crash landing after the commit but before the JSON update would have silently thrown away real, valid work. Recovery logic now checks whether the branch actually advanced past `base_commit` first, and only cleans up if it didn't.
12. **A separate, apply-scoped lock, not the run-cadence lock (Round 3 correction).** Reusing `tools/lib/lock.py`'s existing run lock for implement operations was a mismatch — that lock's staleness TTL is tuned to `max_run_duration_minutes` for scheduled loop runs, and an implement attempt running long could have its lock "recovered" out from under it by a concurrent scheduled run, reintroducing the exact race the lock was meant to prevent. A dedicated apply-scoped lock (same primitive, own name/TTL) avoids the mismatch.

## Risks / open questions

- **Squash-merge/rebase blind spot is accepted, not solved.** If Nate ever squash-merges or rebases a `seo/*` branch before pushing, the exact-SHA ancestry check will never detect it as live, and the proposal will sit in `implemented` (visibly, per decision #10) until he notices and either verifies manually or the `stuck-implemented` flag prompts investigation. Mitigation is operational (merge normally) plus visibility — not a code fix in this phase.
- **MMC collector readiness** — confirmed the registry slot exists, but not confirmed the daily-briefing script actually reads `pending/`/`report.md` today. Treated as a dependency to verify, not assumed.
